Grupo 1: 🔗 URLs mal construidas

El bug más común en frontend. Aparece cuando concatenás strings en vez de armar la URL bien.

1A Fácil

URL duplicada en dbbuilder.html

En el archivo src/static/dbbuilder.html, líneas 215 y 230.

💡 Pista: el "host" aparece dos veces en la misma URL. Mirá bien la concatenación.
let url = `http://127.0.0.1:8000http://127.0.0.1:8000/api/v1/dbbuilder/consultar?db=${nombre}`;
let url = `${window.location.origin}/api/v1/dbbuilder/consultar?db=${nombre}`;

Estado: No resuelto

1B Medio

URL mal concatenada con query params

En dbbuilder.html, función actualizarURLGenerada.

💡 Pista: el operador & para separar params se mete con template literals mal hechos.
let url = '/api/v1/dbbuilder/consultar?db=' + nombre + &limite=&limite; if (campo) url += '&campo=' + campo; // FALTA codificar
let url = `/api/v1/dbbuilder/consultar?db=${nombre}&limite=${limite}`; if (campo) url += `&campo=${encodeURIComponent(campo)}`; // SEGURO

Estado: No resuelto

Grupo 2: ✅ Validación de input

El usuario puede mandar lo que quiera. Si no validás, el server hace cualquier cosa.

2A Fácil

No validar input vacío

En app.py, endpoint /api/v1/dbbuilder/crear.

💡 Pista: si el usuario manda {"nombre": ""}, el server ¿qué hace?
nombre = body.get("nombre", "mi_db") # acepta string vacío
nombre = body.get("nombre", "").strip() if not nombre: raise HTTPException(400, "El nombre no puede estar vacío")

Estado: No resuelto

2B Medio

Tipo de dato incorrecto

En app.py, endpoint /api/v1/dbbuilder/consultar.

💡 Pista: si el usuario manda limite="muchos", el server ¿qué hace?
limite = 100 # el server decide siempre, ignora al usuario
try: limite = int(request.query_params.get("limite", 100)) except ValueError: raise HTTPException(400, "limite debe ser número") limite = max(1, min(1000, limite)) # acotar

Estado: No resuelto

Grupo 3: 💉 SQL Injection

El clásico. Concatenar input del usuario en una query SQL es el error #1 en OWASP.

3A Alto

Concatenar input en SELECT

En app.py, función api_dbbuilder_consultar.

💡 Pista: la query se arma concatenando strings. Un input como ' OR 1=1 -- rompe todo.
sql = f'SELECT * FROM "{db}" WHERE {filtro} LIMIT ?' # INYECCIÓN: filtro = "1' OR 1=1 --" → query rota, filtra TODOS los datos
# Validar filtro contra una whitelist if filtro and not re.match(r'^[a-zA-Z_][a-zA-Z0-9_=<>\'\s]+$', filtro): raise HTTPException(400, "Filtro inválido") # O usar SQLAlchemy con ORM (más seguro)

Estado: No resuelto

3B Alto

Sin prepared statements

En db/db.py, función de insert.

💡 Pista: el INSERT se hace con f-strings, no con placeholders. Cualquier input se cuela.

conn.execute(f"INSERT INTO {tabla} VALUES ({values})") # PELIGRO: values viene del usuario sin sanitizar
cols = ', '.join([f'"{c}"' for c in campos]) placeholders = ', '.join(['?' for _ in campos]) conn.execute(f'INSERT INTO "{tabla}" ({cols}) VALUES ({placeholders})', values) # Placeholders, no concatenación

Estado: No resuelto

Grupo 4: 🔐 Autenticación

El usuario es anónimo hasta que demuestre lo contrario. Si dejás cosas sin auth, son públicas.

4A Medio

Credenciales hardcodeadas

En app.py, en algún lugar de la config.

💡 Pista: las credenciales están escritas en el código fuente. Cualquiera que abre el repo las ve.
DB_PASSWORD = "admin123" # MAL API_KEY = "sk-12345abcde" # MAL
import os DB_PASSWORD = os.environ.get("DB_PASSWORD", "") API_KEY = os.environ.get("API_KEY", "") # Las credenciales van en variables de entorno, no en el código

Estado: No resuelto

4B Alto

Token que no expira

En app.py, sistema de tokens (si existe).

💡 Pista: el token se genera una vez y sirve para siempre. Si te lo roban, no hay forma de invalidarlo.
token = generate_token(user) # El token vive en la DB y nunca se vence # Si lo roban, sirve eternamente
import jwt token = jwt.encode( {"user": user, "exp": datetime.utcnow() + timedelta(hours=1)}, SECRET_KEY, algorithm="HS256" ) # 1 hora de vida, después se regenera

Estado: No resuelto

Grupo 5: 🔍 Information Disclosure

El server cuenta demasiado. Versiones, paths internos, configs. Toda esa info le sirve al atacante.

5A Fácil

Stack trace visible en errores

En app.py, default exception handler.

💡 Pista: cuando algo tira error 500, el server muestra TODO el stack trace. Eso le dice al atacante dónde está el código y qué libs usás.
return JSONResponse(status_code=500, content={"error": str(e), "stack": traceback.format_exc()}) # En producción esto filtra paths internos, versiones de libs, etc.
return JSONResponse(status_code=500, content={"error": "Error interno del servidor"}) # El stack trace va a los logs internos, no al cliente

Estado: No resuelto

5B Medio

Endpoint /debug que filtra config

En app.py, el endpoint /api/v1/debug.

💡 Pista: el endpoint /api/v1/debug está accesible sin auth y devuelve info sensible.
@app.get("/api/v1/debug") def debug(): return {"db_host": "10.20.40.50", "secret_key": "DEMO-REDACTED", "env": "production"} # Shadow API encontrada por el atacante
@app.get("/api/v1/debug") def debug(): if not os.environ.get("DEBUG_MODE"): raise HTTPException(404) return {"status": "ok"} # Solo accesible si DEBUG_MODE está activado en el env

Estado: No resuelto
