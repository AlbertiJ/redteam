¿Cómo funciona?

Cada card es un bug. Te muestra: dónde está (archivo y línea), la pista (qué buscar), pasos concretos (qué hacer para arreglarlo), código MALO (cómo está) y código BUENO (cómo debería estar).

Tu trabajo: abrir el archivo en tu editor, seguir los pasos, cambiar el MALO por el BUENO, guardar, refrescar esta página, y marcar como resuelto.

Tip: probá primero el código MALO en vivo. Por ejemplo, si dice "concatenás mal la URL", andá a la página y mirala romper.

Hay 10 bugs en 5 grupos. Cada grupo tiene 2 bugs + variantes. Empezá por el más fácil.

Tu progreso se guarda en este navegador. Si querés resetear, hacé click en "Resetear progreso".

Grupo 1: 🔗 URLs mal construidas

El bug más común en frontend. Aparece cuando concatenás strings en vez de armar la URL bien. Probalo vos: abrí /static/dbbuilder.html?db=musica y mirá la URL que se genera en la sección "Consultar por URL".

1A Fácil

URL duplicada en dbbuilder.html

En el archivo src/static/dbbuilder.html, función actualizarURLGenerada.

💡 Pista: el "host" aparece dos veces en la misma URL. Mirá bien la concatenación.
📋 Pasos para arreglarlo:
  1. Abrí src/static/dbbuilder.html en tu editor
  2. Buscá la línea que empieza con let url = en la función actualizarURLGenerada
  3. Reemplazá el string hardcodeado http://127.0.0.1:8000 por ${window.location.origin}
  4. Guardá y refrescá la página
  5. Volvé acá y marcá como resuelto
let url = `http://127.0.0.1:8000http://127.0.0.1:8000/api/v1/dbbuilder/consultar?db=${nombre}`;
let url = `${window.location.origin}/api/v1/dbbuilder/consultar?db=${nombre}`;

Estado: No resuelto

1B Medio

URL mal concatenada con query params

En dbbuilder.html, función actualizarURLGenerada.

💡 Pista: el operador + se mete con template literals. Y los valores no se codifican para URL.
📋 Pasos para arreglarlo:
  1. Mismo archivo, misma función
  2. Buscá la línea let url = '/api/v1/dbbuilder/consultar?db=' +
  3. Reescribila usando template literals: `...${nombre}...${limite}...`
  4. Envolvé campo y filtro con encodeURIComponent(...) para que los espacios y acentos no rompan la URL
let url = '/api/v1/dbbuilder/consultar?db=' + nombre + '&limite=' + limite; if (campo) url += '&campo=' + campo; // FALTA codificar
let url = `/api/v1/dbbuilder/consultar?db=${nombre}&limite=${limite}`; if (campo) url += `&campo=${encodeURIComponent(campo)}`;

Estado: No resuelto

Grupo 2: ✅ Validación de input

El usuario puede mandar lo que quiera. Si no validás, el server hace cualquier cosa. Probalo vos: andá a la página del dbbuilder y mirá el formulario "Crear DB Custom".

2A Fácil

No validar input vacío

En src/app.py, endpoint /api/v1/dbbuilder/crear.

💡 Pista: si el usuario manda {"nombre": ""}, el server ¿qué hace? ¿Devuelve error o crea una DB fantasma?
🧪 Probalo vos (manda un POST con nombre vacío): 

        

        

          📋 Pasos para arreglarlo:
          
    
            
  1. Abrí src/app.py
    2. 
            
  2. Buscá @app.post("/api/v1/dbbuilder/crear"
    3. 
            
  3. Después de obtener nombre = body.get("nombre", "mi_db"), agregá: nombre = nombre.strip() if nombre else ""
    4. 
            
  4. Si está vacío, devolvé HTTPException(status_code=400, detail="El nombre no puede estar vacío")
    5. 
            
  5. Probá mandando POST /api/v1/dbbuilder/crear con {"nombre": "", "campos": ["id","a","b"], "datos": []}
    6. 
          

        

        
nombre = body.get("nombre", "mi_db")  # acepta string vacío

        
nombre = body.get("nombre", "").strip()
if not nombre:
    raise HTTPException(400, "El nombre no puede estar vacío")

Grupo 3: 💉 SQL Injection

    
El clásico. OWASP Top 1 histórico. Si concatenás input del usuario en una query SQL, estás regalando tu DB. Probalo vos: mirá este endpoint y pensá qué pasa con el filtro.

    

      

      

        3A
        Alto
      

      
Concatenar input en SELECT

        
En app.py, función api_dbbuilder_consultar.

        
💡 Pista: el query se arma con f-string f"...{input}..." o con +. Cualquier input del usuario debería ir como placeholder ?.

        

          📋 Pasos para arreglarlo:
          
    
            
  1. En app.py, buscá la función que arma la query del SELECT en el dbbuilder
    2. 
            
  2. Buscá el patrón f"SELECT * FROM ... LIKE '%{filtro}%'" o similar
    3. 
            
  3. Reescribila con placeholder: "SELECT * FROM ... WHERE campo LIKE ?" + cursor.execute(query, (f"%{filtro}%",))
    4. 
            
  4. Si tu SQL es dinámico (varios campos), validá el nombre del campo contra una whitelist
    5. 
          

        

        
sql = f"SELECT * FROM (db) WHERE (titulo) LIKE '%{filtro}%' -- // sin escape, vulnerable a 'OR 1=1 --"

        
sql = "SELECT * FROM (db) WHERE (titulo) LIKE ?"
cursor.execute(sql, (f"%{filtro}%",))  // parametrizado, seguro

        
Estado: No resuelto

        Alto
      

      
INSERT sin prepared statements

        
En db/db.py, función de insert.

        
💡 Pista: el INSERT arma los VALUES concatenando strings. Cualquier input del usuario debería ir como placeholder ?.

        

          📋 Pasos para arreglarlo:
          
    
            
  1. En db/db.py, buscá INSERT INTO
    2. 
            
  2. Buscá la concatenación: VALUES ('{valor}') o VALUES ({valor})
    3. 
            
  3. Reescribila con placeholders: VALUES (?, ?, ?) + tupla de valores
    4. 
          

        

        
conn.execute(f"INSERT INTO (tabla) (cols) VALUES ({vals[0]}' + str(cols[1]) + '...")

        
placeholders = ", ".join(["?"] * len(cols))
conn.execute(f"INSERT INTO (tabla) ({cols_quoted}) VALUES ({placeholders})", tuple(vals))

Grupo 4: 🔐 Autenticación

    
El usuario es anónimo hasta que demuestre lo contrario. Si dejás cosas sin auth, son públicas. Probalo vos: abrir /api/v1/auth/admin sin estar logueado.

    

      

      

        4A
        Medio
      

      
Credenciales hardcoded

        
En app.py, en algún lugar de la config.

        
💡 Pista: las credenciales están escritas en el código fuente. Cualquiera que abre el repo las ve. Debería abrir el repo las ve. Debería leerlas de variables de entorno sin default.

        

          📋 Pasos para arreglarlo:
          
    
            
  1. En app.py, buscá PASSWORD, SECRET, API_KEY
    2. 
            
  2. Si ves os.environ.get('DB_PASSWORD', 'admin123') o similar con default hardcoded: es bug
    3. 
            
  3. Reemplazalo por os.environ.get('DB_PASSWORD') SIN default
    4. 
            
  4. Si la variable no está seteada, fallar con mensaje claro al iniciar
    5. 
          

        

        
DB_PASSWORD = "admin123"  # MAL: hardcoded
import os; API_KEY = os.environ.get("API_KEY", "sk-1234abc")  # MAL: con default

        
import os
DB_PASSWORD = os.environ.get("DB_PASSWORD")
API_KEY = os.environ.get("API_KEY")
if not DB_PASSWORD or not API_KEY:
    raise RuntimeError("Faltan variables de entorno")

        4B
        Alto
      

      
Token que no expira

        
En app.py, sistema de tokens (si existe).

        
💡 Pista: el token se genera una vez y sirve para siempre. Si te lo roban, no hay forma de invalidarlo.

        

          📋 Pasos para arreglarlo:
          
    
            
  1. Si no hay sistema de tokens: este bug no aplica (skipealo)
    2. 
            
  2. Si hay: buscá la función que genera tokens (ej: create_access_token)
    3. 
            
  3. Agregá expires_delta=timedelta(hours=1) o lo que corresponda
    4. 
            
  4. En el payload, incluí "exp": datetime.utcnow() + timedelta(...)
    5. 
          

        

        
import jwt; token = jwt.encode({"user": user, "exp": datetime.utcnow() + timedelta(hours=1)}, SECRET_KEY)  # 1 hora de vida, después se regenera

        
import jwt; token = jwt.encode({"user": user}, SECRET_KEY)  # sin exp, válido para siempre

Endpoint /debug que filtra config

        
En app.py, el endpoint /api/v1/debug.

        
💡 Pista: el endpoint /api/v1/debug está accesible sin auth y devuelve variables de entorno sensibles. Solo accesible si DEV_MODE está activado.

        

          📋 Pasos para arreglarlo:
          
    
            
  1. En app.py, buscá @app.get("/api/v1/debug"
    2. 
            
  2. Verificá que esté detrás de un check de variable de entorno: if not os.environ.get("DEBUG_MODE"): raise HTTPException(404)
    3. 
            
  3. En producción, esa variable NO debe estar seteada → endpoint da 404
    4. 
            
  4. Si querés ser más paranoico: que requiera auth + que la respuesta sea mínima (sin secretos)
    5. 
          

        

        
@app.get("/api/v1/debug")
def debug():
    return {"db_host": "10.20.40.50", "secret_key": "DEMO-REDACTED", "env": "production"}  # accesible sin auth

        
@app.get("/api/v1/debug", include_in_schema=False)
def debug():
    if not os.environ.get("DEBUG_MODE"):
        raise HTTPException(404)
    return {"status": "ok"}  # solo si DEBUG_MODE está seteada
